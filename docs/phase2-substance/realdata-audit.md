# Phase 2 Substance Real-Data Audit

## Scope

This audit walks v1 through the same surface area it already has: room creation, invite import, voting, Q&A, synchronization, analytics, and export. It does not propose new features yet. It identifies where the current engine fails when real organizers bring real conference data instead of the curated generated demo.

## Top 5 Logic Gaps

1. **No real roster ingestion.** Organizers arrive with Eventbrite, Zoom, Luma, Google Sheets, or CRM attendee exports; v1 only asks for a count and generates disconnected invites.
2. **No poll draft ingestion.** Real poll questions live in docs/spreadsheets/agendas; v1 ships fixed default polls and has no way to infer poll titles/options.
3. **Brittle invite parsing.** The invite importer accepts exactly one compressed token shape and fails on wrappers humans naturally copy from email, JSON, CSV cells, or URLs.
4. **Broken room links are not recoverable enough.** Some malformed hashes can fail during bootstrap rather than entering an intentional "damaged link" state.
5. **Duplicate cross-device votes are misleading.** The cryptographic nullifier prevents double-counting, but the UI can still tell the second device the vote was published as if it counted.

## Top 3 Intuition Failures

1. **"I have a real attendee list; why am I typing only a number?"** The app ignores the organizer's most important starting artifact.
2. **"I pasted the invite I was sent; why is it invalid?"** The app cannot recognize harmless copy/paste wrappers or line wrapping.
3. **"It said my second vote was published, but the result did not change."** The system is cryptographically correct but narrates the outcome badly.

## Top 3 Feels-Stupid Moments

1. The organizer must manually count/filter attendees instead of the app detecting eligible rows from a roster.
2. The organizer must manually translate poll drafts into the hard-coded v1 poll shape.
3. The attendee must manually clean an invite token copied from a real communication channel.

## What Smart Means For This Product

For `anon-conf-poll`, "smart" means:

1. Pasting or uploading a common attendee roster should immediately produce a useful invite-generation preview: detected rows, eligible voters, duplicates, excluded rows, and confidence.
2. Pasting poll text or CSV should immediately infer poll titles and options, with inline correction rather than configuration-first setup.
3. Pasting an invite should work across common copy shapes: raw token, quoted string, wrapped email text, CSV cell, or invite URL.
4. Damaged room links and invalid manifests should fail in conference-organizer language, not stack/schema language, and always offer a recovery path.
5. Voting/Q&A outcomes should be honest: counted, duplicate, pending verification, or rejected are distinct states that exports also preserve.

## Phase 2 Substance Success Metrics

- Real-data pass rate: at least 7 of these 10 inputs complete their primary flow without manual cleanup beyond a genuine organizer decision.
- Invite normalization: 100% of the tested invite copy shapes either import successfully or identify the exact damage/truncation reason.
- Roster inference: Eventbrite-style and Zoom-style CSV fixtures produce deterministic eligible-attendee previews with duplicate/excluded-row counts.
- Poll inference: at least 2 of 3 poll draft fixtures produce correct poll title/option grouping with no manual configuration.
- Broken-link recovery: 100% of malformed room hashes render a recoverable state; no blank screen or uncaught bootstrap error.
- Duplicate nullifier handling: same invite on two devices never reports the second vote as counted.
- Determinism: same fixture input produces byte-identical normalized output and export metadata, except explicitly versioned generation timestamps.
- Performance: roster/poll parsing preview appears in under 1 second for fixtures up to 5,000 rows; operations over 300 ms expose progress.

## Explicitly Out Of Scope

- No new deployment mode; remain Mode A GitHub Pages.
- No backend, database server, auth system, or organizer account system.
- No visual polish pass, dark mode, command palette, OG images, or new marketing chrome.
- No new voting feature types beyond the existing poll/Q&A surface.
- No moderation, admin roles, payments, or analytics tracking.
- No Phase 3 scale architecture such as owned TURN/signaling relays.
- No change to the core cryptographic protocol unless an ADR later shows a fixture-driven correctness issue.
